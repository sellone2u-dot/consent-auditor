# Dealer Website Risk Auditor v0.2.0 Compatibility Contract

## Architecture Principle

Every user-visible conclusion must be traceable to evidence. Every major feature must be independently testable. Existing v0.1.7 behavior must be preserved unless an intentional behavior change is documented and approved.

## Purpose

This contract defines the compatibility boundary for work leading from v0.1.7 to v0.2.0. It protects the current dealer-facing product while the implementation is made more testable and gains a guided Fresh, Deny, and Accept workflow.

Phase 0 establishes documentation and regression fixtures only. It does not authorize runtime behavior changes or application-logic refactoring.

## Behavior That Must Be Preserved

- The extension name remains **Dealer Website Risk Auditor**.
- The product records observable website behavior and does not certify compliance or provide legal conclusions.
- The existing quick current-page read and stronger reload-based **Scan page** workflow remain available.
- Existing support for CMP, ComplyAuto, GTM, Google, Meta/Facebook, cookies, consent parameters, third-party domains, saved scan states, Markdown export, and printable report export remains intact.
- Page-only ComplyAuto evidence must not be treated as proof that `blocker.js` was captured or that its timing was verified.
- A consent state must not be described as confirmed when no real website Accept, Deny, settings, or banner control was observed.
- Saved Fresh, Deny, Accept, No banner, and Current browser state concepts remain compatible with existing stored snapshots.

## Terminology That Must Be Preserved

- **Fresh visitor / before consent choice**
- **Denied cookies / reject all**
- **Accepted cookies / post-consent**
- **No banner / cannot test Accept-Deny**
- **Current browser state / unknown**
- **Observed Load**
- **Captured request sequence**
- **ComplyAuto Control Evidence**
- **Behavioral Consent Signal Read**
- **High Priority Observations**
- **Observed risk indicators**

The report-facing observed risk ratings remain:

- Strong
- Good / Verify
- Needs Review
- High Risk
- Critical
- Unable to Verify

## Evidence and Language Rules

- Network request observation is evidence of an observed browser request, not proof that downloaded code executed.
- Captured request order must not be presented as proven JavaScript execution order.
- Missing activity means it was not observed in the captured scan; it does not by itself prove blocking.
- Conclusions must identify their supporting observations or explicitly state what evidence was unavailable.
- Derived or heuristic statements must remain distinguishable from direct browser observations and user confirmations.
- Reports must retain dealer-friendly, evidence-based language and the no-legal-conclusions disclaimer.

## Change-Control Rule

An intentional behavior or terminology change requires all of the following before implementation:

1. The change is documented in `BUILD_NOTES.md` or an approved implementation note.
2. The reason and expected user-visible effect are stated.
3. The affected regression fixture or test is updated deliberately.
4. Approval is recorded before the compatibility expectation is changed.

Refactoring alone is not a reason to change established behavior, scoring, terminology, storage meaning, or report conclusions.

## Testability Rule

New major features must expose independently testable logic with browser APIs kept behind narrow integration boundaries. Regression tests must cover preserved v0.1.7 behavior before a related implementation is replaced or reorganized.

The Phase 0 baseline covers:

- manifest identity and local references;
- required compatibility-contract language;
- sanitized request, page, scan, and report fixtures;
- existing report terminology and disclaimer output.

Later phases will add isolated tests for classification, consent signals, request lifecycle, analysis, scoring, comparisons, evidence provenance, storage migrations, and report generation.
